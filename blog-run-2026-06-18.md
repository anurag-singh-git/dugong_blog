## New post

**Dispatch Nº 62 — "The order tag you set by hand"**
`the-order-tag-you-set-by-hand.html` · by Priya Singh · 2026-06-18 · ~1,880 words declared

Chosen because order tagging is a real, high-volume, frequently-complained Shopify pain that no
native feature fully closes, and it was not yet covered by the existing 15 dispatches. It is also the
single best fit for the site's lead essay, whose headline finding is that "tagging is 14% of every
Shopify workflow merchants ship" while the work that scales is under 1%. Order tagging is the clearest
case of that gap: enormous volume, mostly automated badly. The post is distinct from its neighbours:
the high-risk flag (Nº 53) is fraud judgement, the order-change post (Nº 51) is editing an order, and
the bad-address post is shipping validation. This post is the tagging chore itself, the connective
tissue that routes fulfillment and builds segments.

The Shopify-specific limitations are confirmed from current research and form the crux:
(1) Shopify Flow is free on every plan and tags well on a single condition (total over a threshold, a
specific product in the cart), but it checks conditions one at a time, so the multi-signal tags people
set by hand (high value AND slow-ship destination AND repeat customer AND a gift note) become a stack
of overlapping single-condition workflows that are hard to keep in sync. (2) Flow can reference
structured note attributes and line item custom attributes in conditions, but it cannot read a
free-text order note the way a person does, so "please ship before Friday" or "no invoice, it is a
gift" is invisible to the rule, which is the exact signal someone scans the orders list for each
morning. (3) The tags themselves drift and break: nothing keeps the vocabulary clean, so vip / VIP /
Vip can coexist and a segment built on one silently misses the others, and a tag built from a variable
can hit invalid characters or the length limit and throw an "order tags is invalid" error that fails
silently. (4) A related Admin API limit: you cannot change line item properties on an existing order,
only order-level tags and notes.

The angle beyond "install a tagging app": the tag is the cheap part, the reading is the job. Tagging
was never a matching problem, it is a reading-and-judgement problem wearing a matching problem's
clothes. The Dugong playbook reads the whole order at once (line items and properties, note and note
attributes, discount and source, destination, customer history), decides tags from plain-language
rules that combine signals, normalizes every tag against one controlled vocabulary, writes back at the
order level without touching anything else, triggers the routing or hold the tag is for, backfills the
orders that predate the rule so segments are not half-blind, and logs every tag with its reason. A
callout makes the point that a rule engine can apply tags faster than you can but still cannot read.
